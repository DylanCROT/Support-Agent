# Support Agent — Design

**Date:** 2026-05-12
**Status:** Accepted (Phase 0 design)
**Supersedes:** the previous design committed in `38e6627` (the `/daily-triage` orchestrator + `ticket-investigator` subagent model). That work is removed in this refactor.

## Mission

Investigate Salesforce support tickets on the REC board, in status `REFERRED TO PRODUCT OWNER`, assigned to Dylan Crous — one ticket at a time, with Dylan present — and produce a draft Jira comment containing the agent's root-cause hypothesis and supporting evidence. On Dylan's explicit approval, post that comment to the ticket. Nothing else.

This agent is **Salesforce-only**. Tickets originating from other systems are out of scope.

## Scope

### In scope

Three types of REC tickets:

- **Bugs in Salesforce behaviour** — a flow didn't fire, a field is wrong, an automation broke, an unexpected error appeared.
- **Data anomalies** — specific records look wrong (a claim shows $0, an account is duplicated, a field has the wrong value).
- **Rights and permission requests** — a user is asking for access they don't have, or asking why they lost access.

### Out of scope (for now)

The agent classifies these and refuses; Dylan handles them manually.

- Enhancement / change requests
- Bulk update requests
- Process / "why is the system designed this way" questions

### Other boundaries

- The agent reads Salesforce data via the `sf` CLI (read-only SOQL only).
- The agent reads the Salesforce codebase via the `github-salesforce` MCP (read-only).
- The agent's only Jira write is `addCommentToJiraIssue`, gated per-ticket by Dylan's approval. No transitions, no field edits, no reassignment, no issue creation, no linking.

## The per-ticket loop

The agent is invoked with a single Jira URL:

```
/investigate https://vat-it.atlassian.net/browse/REC-1234
```

It then runs this sequence for that one ticket:

1. **Fetch the ticket** via the Atlassian MCP — summary, description, comments, custom fields, reporter, recent activity, links.
2. **Classify** the ticket into one of `bug` / `data-anomaly` / `permission-request` / `out-of-scope`. Present the classification with a one-line rationale and ask Dylan to confirm. Out-of-scope stops the run.
3. **Load the matching playbook** from `playbooks/<type>.md`. This is the procedure the agent will follow.
4. **Investigate.** The agent executes the playbook — running SOQL via `sf data query`, reading code via the `github-salesforce` MCP, checking related tickets. It thinks out loud in markdown so Dylan can see the reasoning trail.
5. **Ask when stuck.** When the agent hits a gap — terminology it doesn't know, a judgment call, missing context — it pauses and asks Dylan. Each answer is captured durably (see Knowledge capture below) before the agent continues.
6. **Draft the comment** to `investigations/<YYYY-MM-DD>/<TICKET-KEY>.md`. Structure: one-line conclusion → evidence cited inline → confidence level → "what I couldn't determine".
7. **Present for approval.** Show Dylan the draft. Ask: **post / edit / discard**. On *post*, call `addCommentToJiraIssue` with the markdown as the body and confirm the comment id. On *edit*, Dylan dictates changes; the agent updates the file and re-asks. On *discard*, do nothing and record the outcome.

The whole loop is one Claude Code session. There is no batch mode in v1.

## Playbook structure

Each of the three playbooks is a single markdown file with this shape:

```
# Playbook: <type>

## When to use
<classification rule — what makes a ticket this type>

## Required inputs
<what the ticket must contain; if missing, draft is "needs info from reporter">

## Investigation steps
<numbered steps; each names a tool and what evidence it captures>

## Common root causes
<bullet list of recurring causes for this type — grows over time>

## Comment template
<skeleton for the final Jira comment for this type>

## Open questions / unknowns
<things the agent has been asked about but isn't sure how to handle — flag during runs>
```

Playbooks start as skeletons in this commit. The "Common root causes" and "Open questions" sections grow each time the agent works a real ticket. The "Investigation steps" section is the most important to flesh out early — it is the playbook's procedure.

## Knowledge capture (the bootstrap mechanism)

When the agent hits a gap and asks Dylan, the answer must be captured durably. Three destinations:

- **`playbooks/<type>.md`** — procedural learnings. *"When investigating bugs, also check recent deploys to the relevant package."*
- **`glossary/<term>.md`** — domain vocabulary. One file per term. *"An RCT is a Reclaim Calculation Task, represented by `Reclaim_Calc_Task__c`, generated by the X flow when Y condition holds."*
- **`examples/<TICKET-KEY>.md`** — worked references from resolved tickets. The ticket body, the investigation, and the comment that resolved it. Built up over time, not authored from scratch.

The agent **proposes** the destination when capturing; Dylan confirms or corrects. Knowledge capture is a deliberate step in the loop, not a side effect of investigation.

## Tool guardrails

The existing `.claude/settings.json` deny-list already covers:

- All mutating `sf` subcommands (`create-record`, `update-record`, `delete-record`, `delete`, `upsert`, `import`, `apex run`, `project deploy`, `project delete`, `org delete`) — under both Bash and PowerShell invocations.
- Atlassian MCP mutations beyond `addCommentToJiraIssue`: `createJiraIssue`, `editJiraIssue`, `transitionJiraIssue`, `createIssueLink`, `addWorklogToJiraIssue`, and all Confluence writes.

That stays. Hard rules also live in `CLAUDE.md` so the agent is prompt-level aware, not only blocked at the permission layer.

## Audit trail — `framework-log/`

This folder exists to make the *build process itself* reusable. Future agent projects can follow the same phases.

1. **`decisions/`** — ADRs (Architecture Decision Records). One file per material design decision. Format: context → decision → alternatives considered → consequences. Numbered sequentially.
2. **`sessions/`** — one file per working session. Captures: what we set out to do → what we discussed → what we decided → what we left open → files created/deleted.
3. **`README.md`** — the meta-framework. Describes the reusable phases (brainstorm → spec → refactor → plan → bootstrap → distil → steady-state) and how to apply them to a new agent project.

`PROGRESS.md` at the repo root is the operational tracker — a phase checklist, updated each session. The framework-log is the historical record; PROGRESS.md is "where are we now."

## Build phases

This design covers Phases 0–4. Phase 5 (batch, scheduled runs, new ticket types) is explicitly out of scope.

| Phase | Goal | Exit criterion |
|-------|------|----------------|
| 0 — Design | Agree shape and decisions | Spec committed; user approves |
| 1 — Scaffolding | `/investigate` works end-to-end on a real ticket | One successful dry-run with Dylan |
| 2 — Bootstrap | Run on ~10 real tickets; capture knowledge | Each playbook has ≥3 entries in "Common root causes" and ≥2 referenced glossary terms |
| 3 — Distil | Prune incidental knowledge; harden playbooks | Playbook diffs stabilise; `examples/` library has ≥5 entries |
| 4 — Steady state | Agent runs unaided on most tickets | 80%+ of runs complete without Dylan being asked mid-investigation |

## What this design deliberately does not do

- **No batch / queue mode.** Single-ticket-by-URL only.
- **No automation of Jira state beyond commenting.** No transitions, no reassignment.
- **No shared knowledge import from po-assistant or anywhere else.** Knowledge lives in this repo.
- **No subagents.** The previous design used a `ticket-investigator` subagent; we removed it. One Claude Code session, one ticket, in-process.
- **No code edits to the Salesforce repo.** Read-only, always.
- **No coverage of enhancement / bulk-update / process-question tickets.** Classified and refused.
